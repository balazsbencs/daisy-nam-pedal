01 · Build

Build from a clean clone.

The CI reference environment is Ubuntu with Arm GNU Toolchain 15.2.Rel1. Local builds need an arm-none-eabi-gcc toolchain, GNU Make, Git, Python 3, and dfu-util for flashing.

git clone --recurse-submodules https://github.com/balazsbencs/daisy-nam-pedal.git
cd daisy-nam-pedal
tools/apply_submodule_patches.sh
make -C third_party/libDaisy
make -C third_party/DaisySP
make

Build products are written to build/NamPlatform.elf, .hex, and .bin. The application is roughly 600 KiB and cannot fit in the STM32H750's 128 KiB internal flash.

BOOT_QSPI is mandatoryAPP_TYPE = BOOT_QSPI in the root Makefile makes the application execute from external QSPI. Do not switch this project to BOOT_SRAM or an internal-flash target.

03 · Runtime boundaries

Keep work out of the audio IRQ.

Audio callback · every 1 ms

Deterministic DSP only

Read input, sample click-less encoders, process tuner or audio chain, copy mono output to L/R, record peak cycle count.

Main loop

Control and allocation

Debounce gestures, load/free models and IRs, write presets with audio stopped, render display frames, print diagnostics.

The callback receives 48 samples at 48 kHz: exactly a 1 ms deadline. Model objects and IR convolvers are constructed in the main loop, then atomically exposed to the callback. Never allocate, erase flash, print, or block in the callback.

Model reload deliberately swaps the active model to null before allocating the replacement. This creates a brief dry path but frees the fast-SRAM arena; retaining both models would spill the new arena into SDRAM and miss the audio deadline.

04 · DSP

Audio path and supported models.

INPUT→GAIN→GATE→COMP→NAM→IR→EQ→DELAY→VOLUME

The current build enables real-time effects and the optimized A2 path. It defines NAM_ENABLE_A2_FAST, NAM_A2_NANO_ONLY, and disables slimmable WaveNet. Treat the firmware as supporting the embedded A2 Nano path—not arbitrary desktop .nam architectures. The data packer accepts binary .namb blobs.

Bypass is an early dry copy. Tuner mode is another early path: it pushes dry input into TunerDetector, mutes both outputs, and skips all amp/effect processing.

05 · Main-loop state

Controls and UI flow.

PERFORMANCEclick Gain →BROWSEhold Gain →EDITTUNER
both switches
  • Controls::Process() emits high-level taps, holds, encoder steps, and the both-switch chord.
  • Encoders 2–5 are sampled at the 1 kHz audio-block rate so display transfers cannot drop detents.
  • Ui renders into a framebuffer and transfers up to 16 display rows per main-loop call, capped at 30 fps.
  • Only one of browse, edit, or tuner is active; performance is the default when all are false.
06 · Flash ABI

QSPI memory layout.

RangeSizeUse
0x90000000–0x9003FFFF256 KiBDaisy QSPI bootloader.
0x90040000–0x901FFFFF1.75 MiB reservedExecute-in-place application region.
0x90200000–0x907FFFFF6 MiBModel, IR, and preset data partition.

data_format.h is the firmware ABI: an 8-byte NamDataHeader, 48-byte directory entries, then 4 KiB-aligned blobs. Models contain raw NAMB bytes; IRs contain little-endian mono float32 taps; presets are packed 138-byte NamPreset records that reference model and IR names.

The running firmware reads data memory-mapped. A preset save stops audio, disables interrupts, erases its aligned 4 KiB sector, writes the record, restores interrupts, then restarts audio. Model/IR data replacement still uses DFU.

07 · Assets

Build and inspect a data image.

From the repository root, create a folder containing binary models, WAV impulse responses, and an optional presets.json:

mydata/
├── be100.namb
├── friedman_212.wav
└── presets.json
[
  {
    "name": "BE100 Lead",
    "model": "be100",
    "ir": "friedman_212",
    "input_gain": 1.0,
    "output_volume": 0.8,
    "bypass": false,
    "eq": {"bass_gain": 2, "mid_gain": -1, "treble_gain": 1,
           "bass_freq": 100, "mid_freq": 750, "treble_freq": 4000}
  }
]
python3 tools/build_data_image.py mydata -o data_image.bin
python3 tools/inspect_data_image.py data_image.bin

The packer needs only Python's standard library. It downmixes supported PCM/float WAV files to mono, trims to 256 taps by default to match the convolver, peak-normalizes the frequency response to a 0 dB ceiling, and rejects an image larger than 6 MiB. Use --max-taps 0 only after changing and validating the firmware convolver limit.

Name matching is exactPreset model and ir values must match file stems after the packer truncates names to 30 ASCII characters. An empty IR name means cab bypass.
08 · DFU

Flash bootloader, app, and data.

Enter STM32 DFU by holding BOOT, tapping RESET, then releasing BOOT. Install the QSPI bootloader once:

tools/flash_app.sh --boot

The board reboots. Enter DFU again, then build and flash the application at 0x90040000:

tools/flash_app.sh

Build the data image, enter the DaisyBoot DFU window as prompted, then write the data partition at 0x90200000:

tools/flash_data.sh data_image.bin
Do not swap the addressesApplication and data are separate QSPI regions. Writing a data image at the app address destroys firmware; writing firmware at the data address destroys models, IRs, and presets.
09 · USB CDC

App-triggered bootloader entry.

The firmware listens on internal USB CDC for one exact newline-terminated command:

NAM_DFU_BOOT 7E57A11E

BootloaderCommandParser accepts fragmented USB packets, resets overlong lines, ignores non-matching text, and calls the Daisy system reset-to-bootloader path only on an exact line match. A host tool may send this command instead of requiring physical BOOT/RESET access; the USB serial device name is host-specific.

10 · Verification

Tests, CI, and releases.

  • Firmware PR CI: recursively checks out submodules, applies maintained patches, caches libDaisy/DaisySP, builds with Arm GNU 15.2.Rel1, and prints ELF memory use.
  • Release workflow: derives semantic versions from conventional commits, builds firmware, and packages versioned BIN/ELF artifacts.
  • Website CI: checks static links/structure and simulator transitions before deploying GitHub Pages.
Known baseline issueAt this revision, tests/Makefile contains committed merge-conflict markers and the aggregate host suite stops with “missing separator.” Resolve that file before using make -C tests run; this website change intentionally does not alter unrelated firmware test infrastructure.

Individual test sources remain useful references for QSPI, presets, DSP, controls, UI mode, display transfer, and tuner behavior.

11 · Maintained patches

Keep upstream pins clean.

NeuralAmpModelerCore, nam-binary-loader, libDaisy, and DaisySP are pinned submodules. Repository-owned embedded changes live as patch files under patches/submodules/, not as undocumented dirty submodule commits.

git submodule update --init --recursive
tools/apply_submodule_patches.sh

The patch script is idempotent. Before updating a submodule pin, read Maintained Submodule Changes for each patch's rationale, validation, and retirement condition.

12 · Change safely

Contribution checklist.

  1. Start from a clean clone with recursive submodules and applied patches.
  2. Keep HardwareConfig.h, data_format.h, and the desktop packer in sync when changing pins or storage ABI.
  3. Keep allocation and blocking work out of AudioCallback.
  4. Add the smallest host check that fails before and passes after non-trivial logic changes.
  5. Build firmware and inspect arm-none-eabi-size build/NamPlatform.elf.
  6. Use conventional commit prefixes so release automation can classify the change.
